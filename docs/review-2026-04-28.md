# PDF-RAG-Agent V4 架构审视与重构建议

> 日期：2026-04-28
> 角色：Agent 架构师视角
> 目标：从"巨型有限状态机式 RAG 流水线"演进为"通用 Tool-Use Loop 智能 Agent"，对齐 Claude Code / OpenAI tool-use 主流范式。

---

## 1. 执行摘要（TL;DR）

通读 `app/services/agent.py`（**7400 行**）、`app/services/agent_mixins/*`（共 **5798 行**）、`app/services/intent.py`（1172 行）、`app/services/agent_runtime.py`、`agent_planner.py`、`agent_tool_registries.py`、`agent_tools.py` 后，识别出 **8 个最关键问题**：

1. **巨石单体 Agent**：`ResearchAssistantAgentV4`（`@/home/ubuntu/owen/pdf-rag-agent-v4/app/services/agent.py:84-7400`）通过 6 个 Mixin 拼出了一个 7400+5798≈**1.3 万行**的"上帝类"，违反 SRP，难以演进。
2. **Tool 系统其实是"动作枚举 + 强约束注册器"，不是真正的 Tool-Use Loop**：仅 6 个固定原子动作（`read_memory / search_corpus / web_search / query_library_metadata / compose / ask_human`，见 `agent_tools.py:55-100`），且 `_openai_tool_definitions`（`model_clients.py:250-280`）把每个工具的 `parameters` 写死成只有 `reason` 字段——**LLM 无法真正用结构化参数驱动工具**，整个系统退化成"LLM 选个名字 → Python 写死的处理函数干活"。
3. **重度硬编码关键词路由**：`intent.py` 与 `agent.py` 中遍布 `markers = [...]` 中文短语数组（`intent.py:909/933/951/1041-1076`、`agent.py:1244/2034-2068/3320-3382/4044-4083/5491-5503` 等数十处），构成事实上的规则引擎，与"LLM 真正思考"目标背道而驰。
4. **Intent → Contract → Plan → Tool sequence 多层硬编码串联**：`research_tool_sequence`（`agent_tools.py:126-136`）、`_solver_sequence_for_goals`（`agent.py:5740-5782`）、`_research_plan_goals`（`agent.py:5814`）形成"goal 集合枚举 → solver 序列枚举"的固定流水线，新问题类型必须改代码。
5. **缺乏真正的"思考/不确定性"机制**：没有 logprobs、没有 self-consistency、没有 verifier-guided sampling；`ask_human` 触发条件由 `_contract_needs_human_clarification`（`agent_runtime.py:294`）等基于 contract 字段的硬规则决定，而非基于置信度。
6. **`compose` 工具是黑盒终点**：根据 goal 集合分发到 `_solve_origin_lookup_text` / `_solve_entity_definition_text` / `_solve_topology_discovery_text` 等 **15+ 个写死的 solver**（`solver_pipeline.py:308-955`），每种问法都要新写一个 solver——**与"通用 Agent"理念完全相反**。
7. **流式仅在最外层 SSE，工具内部不流**：`astream_chat_events`（`agent.py:168`）走 `_run` 同步管线后包成事件；缺少 token-level streaming 与 tool_use / tool_result 增量推送。
8. **没有"Agent 自拓展"机制**：工具集编译期固定（`AGENT_TOOL_SPECS = (...)`），无运行时注册；无 learnings/memory 持久化注入 system prompt（仅有 `app/prompts/assistant_self.md` 22 行静态身份描述）。

**总改造方向**：用 **Claude Code 风格的 tool-use 主循环 + 真正带 JSON Schema 参数的原子工具集 + 动态注册 registry + 基于置信度的 ask_human + learnings 持久化** 替换当前"硬编码状态机 + 动作枚举"。落地分 M1 / M2 / M3 三步，**不必一次重写 7400 行**。

### 3.3 Tool 参数被强制为 `{reason: string}`

**改法**：每个工具自带 `input_schema: dict`（OpenAI / Anthropic 通用 JSON Schema），`_openai_tool_definitions` 直接透传 `tool["input_schema"]`。

### 3.4 流程编排硬编码

- `agent.py:211-457` `_run`：在 LLM 驱动之前，**Python 端先用 if 链**判定 compound / citation_ranking / memory_synthesis / library_status 等若干"前置短路"。这违反"Agent 自己思考"的核心要求。

- `prompts/tool_use_protocol.md`（tool-use 主循环说明）

## 4. Agent 主循环重构方案（Claude Code 风格）

### 4.1 主循环骨架（替换 `agent.py:211 _run` + `agent_runtime.py:78`）

```python
# app/services/agent_loop.py（新建，~200 行，取代当前 _run + run_research_agent_loop）
async def run_agent_loop(
    *, query: str, session: SessionContext, ctx: AgentContext, emit: Emit
) -> FinalAnswer:
    messages: list[Message] = build_initial_messages(query, session, ctx)
    for step in range(ctx.settings.max_agent_steps):  # 默认 16
        async for delta in ctx.llm.astream(
            system=ctx.system_prompt,        # core + learnings + tool protocol
            messages=messages,
            tools=ctx.tool_registry.schemas(),  # 真正的 input_schema
            temperature=ctx.settings.temp,
            logprobs=True,                   # 用于 confidence
        ):
            emit(delta.event, delta.payload)  # token / tool_use / thinking
        assistant_msg = delta.final_message
        messages.append(assistant_msg)
        if assistant_msg.stop_reason == "end_turn":
            return FinalAnswer.from_message(assistant_msg)
        # 执行 tool_use 块（可并行）
        tool_results = await asyncio.gather(*[
            ctx.tool_registry.run(call, session=session, emit=emit)
            for call in assistant_msg.tool_uses
        ])
        messages.append(tool_results_message(tool_results))
        # 低置信度 → 注入反思提示（不强制 ask_human，由 LLM 决定）
        if assistant_msg.confidence < ctx.settings.confidence_floor:
            messages.append(reflection_hint(assistant_msg))
    return FinalAnswer.timeout()
```

## 7. 思考与不确定性

### 7.1 置信度三种来源

1. **logprobs 平均**：`model_clients.py:96 stream_text` 增加 `logprobs=True`，输出 token 级 logprob，对最终 answer 段取均值或最小值。
2. **self-consistency**：同一问题 LLM 采样 N=3 次，answer 主旨向量相似度方差。
3. **verifier**：用 cheap 模型做"答案 vs evidence"NLI 判定（已部分存在 `claim_verifier.py`）。

```python
@dataclass
class Confidence:
    score: float           # 0..1
    basis: str             # "logprobs"|"self_consistency"|"verifier"
    detail: dict
```

### 7.2 触发 ask_human 的"软规则"（替换 `agent_runtime.py:294 _contract_needs_human_clarification`）

```python
def should_ask_human(conf: Confidence, settings) -> bool:
    return conf.score < settings.confidence_floor   # 唯一硬阈值，且来自 config
```

且**最终是否问**仍由 LLM 决定（在 system prompt 中告知"低置信时优先调用 ask_human"），不在 Python 端强制中断。

### 7.3 Reflection / Critique 步骤

每收到 `tool_result` 后注入轻量 reflection 提示（仅在 `confidence<floor` 时），避免在简单 turn 上无谓增加 latency。

### 13.2 主循环伪代码（精简版）

```python
async def run_agent_loop(query, session, ctx, emit):
    msgs = [user(query)]
    for _ in range(ctx.max_steps):
        rsp = await ctx.llm.acomplete(system=ctx.sys, messages=msgs,
                                      tools=ctx.registry.schemas(), stream=True, logprobs=True)
        async for d in rsp.stream(): emit(d.event, d.payload)
        msg = rsp.final
        msgs.append(msg)
        if msg.stop_reason == "end_turn": return msg
        results = await asyncio.gather(*[ctx.registry.run(c, ctx=ctx, session=session) for c in msg.tool_uses])
        msgs.append(tool_results_message(results))
    return timeout_message()
```
